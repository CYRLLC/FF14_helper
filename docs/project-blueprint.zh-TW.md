## 1. 文件目的

這份文件用來整合 FF14 Helper 目前尚未完成、尚未開始、以及未來預計執行的所有主要工作，避免藍圖散落在對話、README、個別 issue 或臨時想法中。

本文件的目標不是描述單一功能，而是建立一份可持續維護的執行框架，讓之後每一次改版都能回答以下問題：

- 目前網站已經做到哪裡。
- 哪些功能只是「可用」，但還遠不到成熟。
- 哪些部分是明確的產品方向，不應再反覆搖擺。
- 哪些部分暫時不做，避免範圍失控。
- 接下來應該先做什麼，後做什麼，為什麼。

## 2. 專案定位

FF14 Helper 的定位不是單一工具頁，而是一個以台灣玩家工作流為核心的 FF14 前端工具站。

目前站內已經涵蓋的主題包含：

- 個人設定備份與還原檢查
- 繁中服查價工作台
- 金碟 GATE 時段與候選預測
- 藏寶圖點位與多人規劃
- 製作助手
- 收藏追蹤
- 工具箱與來源整理

這個專案的核心產品原則如下：

1. 以前端為主，不依賴本站後端。
2. 以繁體中文與台灣玩家常用流程為優先。
3. 能站內化的功能盡量站內化，但必須清楚標示參考來源。
4. 對外部開源專案採「研究、重製、整合」策略，而不是直接照搬 UI 或內容。
5. 對資料來源、短暫儲存、第三方 API、授權邊界必須公開透明。

### 3.3 已完成但仍未成熟的能力

以下項目已存在實作，但仍有明顯的品質落差或產品深度不足：

- 備份流程可用，但缺少更完整的還原導引與雲端備份管理。
- 查價頁可做 OCR、手動比價與試算，但整體仍偏工作表，而不是成熟的查價工具。
- 金碟頁已有時段與候選預測，但預測模型與 UI 表達仍很初階。
- 藏寶圖頁已具備多人房間與點位資料，但互動節奏、隊伍管理、視覺密度仍未達成熟工具水準。
- 製作頁已有屬性、配方、模擬、solver、macro，但距離 BestCraft 等級仍有不小差距。
- 收藏頁有追蹤邏輯，但在日常使用上的導向性與可視性仍不足。

### 3.4 已確立的方向

以下產品方向已經明確，不應再回到模糊狀態：

- 查價功能維持「繁中服工作流」而非全球市場聚合。
- 藏寶圖多人協作允許第三方短暫儲存，但只存房間與路線資料。
- 站內所有重要功能都要盡量繁體中文化。
- 來源標示不是附加項，而是全站規範。
- BestCraft、collection-tc、FFXIV Market、treasure-finder 等參考站，皆採重新實作策略。

## 4. 主要問題診斷

這一章只描述目前還沒做完、或做得不夠好的地方。

### 4.1 全站層級問題

#### 4.1.1 文案與語系一致性尚未徹底完成

雖然主要頁面已逐步改成繁中，但仍存在下列風險：

- 某些細節文案仍偏工程語氣，不像公開工具站。
- 部分欄位名稱、說明與錯誤訊息仍不夠自然。
- 有些功能使用的是中文，但內部術語和外部參考站常用語尚未完全對齊。

#### 4.1.2 視覺系統還不夠成熟

目前網站已可用，但視覺層面仍明顯帶有「工程樣板 + 功能堆疊」感。

待改善項目：

- 大螢幕下的欄位與內容密度分配仍可再優化。
- 不同頁面之間的節奏雖已統一，但層次感還不夠強。
- 較複雜頁面仍容易形成長頁滾動負擔。
- 高資訊密度模組缺少固定的操作區、結果區、摘要區節奏。

#### 4.1.3 資訊架構需要進一步收斂

雖然主導覽已縮減，但站內模組持續成長後，仍有以下風險：

- 首頁將逐漸變成所有功能的堆疊入口。
- 次要頁面與實驗性頁面仍需要更清楚的定位。
- 工具箱、站內工具、外部入口之間的界線需要更穩定的規則。

### 4.2 查價頁問題

查價頁目前最大的問題不是不能用，而是使用者認知和頁面本體仍有落差。

目前已存在：

- OCR 匯入
- 批次貼上
- 手動輸入
- 陸行鳥 / 莫古力雙服比價
- 市場板試算
- 最新匯入摘要

但尚未成熟的地方包含：

- 主工作流還不夠集中，初次使用者仍容易不知道從哪裡開始。
- OCR 預覽雖存在，但校對與提交的回饋還不夠強。
- 最近匯入與最近變更區塊還能更像「工具面板」，而不是單純資訊列。
- 批次操作與大量清單整理體驗仍偏弱。
- 缺少更明確的資料新鮮度說明與使用情境提示。
- 尚未建立針對繁中服玩家常見材料、清單、模板的工作流。

### 4.3 藏寶圖頁問題

藏寶圖是目前最有機會成為站內代表性工具的頁面，但也最容易因為功能多而變複雜。

目前已存在：

- 單人 / 8 人分流
- 點位資料
- 地圖預覽
- 房間建立與加入
- Firebase 即時同步
- 路線整理
- 座標複製

但尚未成熟的地方包含：

- 房間管理區仍可更集中、更像成熟協作工具。
- 8 人模式的隊伍管理、路線分配與即時狀態呈現仍不夠強。
- 單人模式與 8 人模式雖已分流，但視覺上還能更清楚地做成兩條不同流程。
- 點位卡、預覽卡、隊伍卡之間的互動關係還不夠直覺。
- 缺少房間過期、同步失敗、離線 fallback 的明確 UX。
- 手機版與大螢幕版可以再做更不同的版型策略。

### 4.4 製作頁問題

製作頁目前最接近「已經是一個工具」，但和原始參考站相比仍存在本質差距。

目前已存在：

- 各製作職業獨立能力值
- 配方搜尋與自動帶入
- 技能面板
- 技能序列
- 模擬結果
- solver
- macro 匯入與輸出
- 與老主顧 / 友好部落的任務關聯

但尚未成熟的地方包含：

- 與 BestCraft 相比，工作台節奏仍未完全對齊。
- 技能規則雖已擴充，但完整度還不夠。
- solver 的品質與可靠度仍需要更多調教。
- 配方、技能、結果、macro 的版面分區仍可更清晰。
- 大螢幕下應更像專業工作台，而不是單一路徑長頁。
- 手機版需要不同的資訊折疊策略。
- 缺少更完整的數值驗證與高等配方實測校正。

### 4.5 收藏頁問題

收藏頁的主要問題不是資料量不足，而是操作回饋不夠有「每日 / 每週使用」感。

目前已存在：

- 老主顧與友好部落清單
- 搜尋、篩選、願望清單、狀態切換
- 匯出 / 匯入
- 焦點模式

但尚未成熟的地方包含：

- 缺少真正可執行的 checklist 體驗。
- 缺少「今天先做什麼」「本週還剩什麼」這類高價值提示。
- 清單卡片的層次與摘要還可以更清楚。
- 與製作頁、查價頁之間的串接還很弱。

### 4.6 備份與還原問題

目前備份與還原屬於可用階段，但不是完整的資料管理工具。

待補項目：

- 還原教學與風險提示
- 雲端備份清單
- 檔案版本比較
- 備份命名與備份摘要管理
- 常見錯誤排除說明

## 5. 核心執行原則

後續所有開發都應遵守以下原則。

### 5.1 先修工作流，再加功能

只要某一頁的主流程還不夠順，就不應急著疊更多邊角功能。這在查價頁、藏寶圖頁、製作頁尤其重要。

### 5.2 先收斂視覺節奏，再追求頁面數量

站內功能數量已經不少，接下來應優先把主功能頁做成熟，而不是再擴張大量次要工具。

### 5.3 明確區分三種資料

1. 只存在本機瀏覽器的資料  
2. 直接送到第三方服務的資料  
3. 短暫存在第三方即時資料庫的協作資料

每一頁都要能清楚說明自己屬於哪一種。

### 5.4 來源標示必須隨功能一起成長

任何受外部工具啟發的新能力，都要同步補來源、說明、重製聲明、授權邊界。

### 5.5 不引入有 ToS 風險或不可控的功能

本站不碰：

- 遊戲自動化
- 外掛注入
- 記憶體讀寫
- 封包攔截
- 偽裝成官方資料來源的功能

## 7. 近期未執行工作

以下項目屬於「已明確需要做，但尚未完成」的近期工作。

### 7.1 全站視覺與文案整頓

#### 目標

把目前站內所有主頁整理到公開工具站等級，而不是功能展示頁。

#### 工作項目

- 全站文案盤點與統一
- 主要區塊命名規則統一
- 錯誤訊息與空狀態重寫
- 大螢幕與手機版分流優化
- 首頁 hero 與主功能卡重設
- About 頁改成公開專案說明頁，而不是附錄頁

#### 驗收標準

- 任一主要頁面在第一次進入時都能清楚知道主流程。
- 大螢幕不再呈現「單欄長頁 + 大量留白」。
- 手機版不需要來回跳區塊才能完成主流程。
- 主要頁面用語一致，不再有明顯工程稿語氣。

### 7.2 查價頁重構

#### 目標

把查價頁從「比價工作表」提升成「繁中服查價助手」。

#### 工作項目

- 重設首頁區塊，明確第一步就是貼圖 / 上傳 / 貼文字
- OCR 預覽卡增強：逐列編輯、刪除、目標服切換、合併策略
- 最近匯入資料與最近變更改成更清楚的摘要面板
- 工作表支援更好的排序、批次刪除、批次更新
- 常用清單模板
- 常用材料快速輸入
- 以用途導向整理試算區，而不是孤立區塊
- 增加資料說明：本站沒有提供官方即時市場 API

#### 驗收標準

- 新使用者能在 30 秒內知道如何開始匯入資料。
- OCR 匯入後可以不離開當前區塊完成校對與提交。
- 比價表能快速找出較便宜的伺服器與差價。
- 試算與比價結果的關聯更直接。

### 7.3 藏寶圖頁重構

#### 目標

把藏寶圖頁做成站內最成熟的多人協作工具之一。

#### 工作項目

- 單人模式重新簡化，聚焦在地圖、點位、座標、收藏
- 8 人模式改成房間導向 UI
- 房間面板整併：房名、房號、連結、TTL、在線成員、同步狀態
- 路線分配、完成標記、成員指派更強化
- 地圖預覽區與路線區同步高亮
- 邀請流程與入房流程更清晰
- 離線 fallback 流程寫清楚
- 房間過期與同步錯誤 UX 補齊
- 手機版針對 8 人模式做簡化資訊分頁

#### 驗收標準

- 8 人模式建立房間後，隊友可直覺加入。
- 使用者可分辨「本地規劃」與「即時房間」差異。
- 點位、路線、隊員、地圖預覽之間的關聯清楚可見。
- 房間過期或同步失敗時不會讓整頁失去可用性。

### 7.4 製作頁深化

#### 目標

把製作頁提升到接近 BestCraft 的站內替代級別。

#### 工作項目

- 重新整理大螢幕雙欄工作台
- 配方搜尋、屬性設定、技能序列、模擬結果、macro 區塊固定節奏
- 每個職業獨立 rotation 草稿
- 更完整的 action 規則與條件
- solver 評分策略調校
- 結果視圖強化：每步明細、buff 狀態、CP / 耐久變化
- 更完整的 macro 輸出與貼上回流
- 針對高難度配方做校正用 preset

#### 驗收標準

- 使用者能清楚理解目前職業、配方職業與屬性之間的關係。
- 大螢幕下能同時看技能、序列、結果，不需頻繁捲動。
- solver 產出的序列具備穩定可讀性。
- macro 內容與模擬結果的對照更直接。

### 7.5 收藏頁深化

#### 目標

把收藏頁做成真正有日常使用價值的追蹤工具。

#### 工作項目

- 每日 / 每週視角
- 建議下一步面板
- 常做內容 pin 起來
- 與製作 / 查價頁串接
- 更好的完成狀態視覺
- 更清楚的老主顧 / 友好部落分流

#### 驗收標準

- 使用者打開頁面時，能立即知道今天和本週最適合先做什麼。
- 收藏頁不只是資料庫，而是行動導向頁面。

## 9. 長期藍圖

以下項目屬於長期方向，不應影響近期主線，但值得持續保留。

### 9.1 站內知識層

建立「不依賴即時 API、但對台灣玩家有價值」的站內知識層，例如：

- 任務導向清單
- 日常 / 每週工作流整理
- 製作與採集實用模板
- 常見活動說明與注意事項

### 9.2 更成熟的協作工具

藏寶圖多人房間若被證明使用頻率高，可長期演進為：

- 更好的路線協作
- 房間歷時摘要
- 更豐富的成員顯示
- 自動過期政策與狀態監控

### 9.3 製作助手超越 BestCraft 的方向

若要以「超越」為目標，路線應該是本地化與工作流整合，而不是單純抄 solver：

- 更適合繁中玩家的說明與流程
- 和查價 / 收藏 / 任務工作流連動
- 更容易上手的 preset 與任務模板
- 更清楚的數值與步驟解釋

## 10. 技術藍圖

### 10.1 前端架構原則

- 維持 React + TypeScript + Vite
- 頁面層與模組層分離
- 共用型別與共用文案集中管理
- 站內儲存優先用 localStorage，必要時加入版本管理
- 可快照化的第三方資料，優先改成本地版本化資料檔

### 10.2 資料來源策略

#### 優先順序

1. 本地靜態資料
2. 可驗證且穩定的公開 API
3. 使用者自行輸入 / OCR / 匯入
4. 第三方即時資料庫短暫儲存

#### 原則

- 避免執行期解析第三方網站腳本。
- 任何第三方 API 都要有失敗 fallback。
- 若資料無法保證正確性，就要明講只是參考。

### 10.3 即時同步策略

目前可接受的即時同步範圍：

- 房間名稱
- 房間代碼
- 成員暱稱
- 路線資料
- 時間戳

目前不應擴張到：

- 永久帳號
- 個人檔案
- 跨功能的長期雲端個人資料

### 10.4 測試策略

#### 單元測試

- 查價計算與 OCR 解析
- 藏寶圖路線與座標工具
- GATE 時間計算
- 製作模擬核心規則
- 收藏儲存與匯出 / 匯入

#### 整合測試

- 備份打包與還原檢查
- 即時房間建立 / 加入 / 更新
- 查價主工作流
- 製作配方搜尋到 macro 輸出的完整流程

#### 人工驗收

- GitHub Pages 真機操作
- Chrome / Edge
- 大螢幕與手機版
- 台灣時區與非台灣時區的金碟頁驗證

## 11. UI / UX 藍圖

### 11.1 全站版型策略

- 主要功能頁採大螢幕工作台式版型
- 次要資訊頁採閱讀型版型
- 手機版不盲目照搬桌機雙欄，必要時改為分段流程

### 11.2 主功能頁共通節奏

任何高價值工具頁都應盡量遵守以下順序：

1. 頁面定位與資料說明
2. 主操作區
3. 即時摘要
4. 詳細結果 / 清單 / 模擬
5. 來源與資料政策

### 11.3 可用性規範

- 第一屏就看得到主行動按鈕或主輸入區
- 重要的狀態變化需要有即時回饋
- 空狀態要能引導下一步
- 每頁需明確說明本站是否保存資料、是否呼叫第三方服務

## 12. 授權、來源與公開專案治理

### 12.1 必須持續維護的內容

- 每個主要功能頁的來源區塊
- README 中的 Reference Sources
- README 中的 Data Policy
- 重新實作聲明

### 12.2 對外部開源專案的處理原則

- 可以研究工作流
- 可以研究資料結構
- 可以研究演算法方向
- 不直接複製 UI、文案、素材
- 若引用語系、資料或授權資訊，需明確標註

### 12.3 素材策略

若未來需要圖片、圖示或背景素材：

- 優先使用開源或可明確授權的素材
- 將來源記錄進 README 或頁面來源區
- 避免直接使用遊戲受限資產而未說明來源

### Milestone A：基礎可用性收斂

目標：把全站主頁面提升到穩定可公開展示的水準。

- 大螢幕與手機版關鍵頁修整
- 主流程空狀態與錯誤狀態修補

### Milestone B：查價與藏寶圖成熟化

- OCR 預覽與工作表深化

### Milestone C：製作助手深度化

目標：讓製作頁接近 BestCraft 等值替代。

範圍：

- 規則補全
- solver 調整

- 備份 -> 還原 -> 雲端管理

### Milestone E：公開專案成熟化

目標：讓專案更適合長期公開維護。

## 14. 不做清單

為了避免範圍失控，以下項目目前明確不做：

- 遊戲自動化

- 記憶體讀寫
- 封包分析
- 需要自建後端與永久帳號系統的功能
- 假裝提供不存在的官方或可靠市場 API
