最後更新：2026-03-25

---

## 1. 文件目的

這份文件專門回答一個問題：

> FF14 Helper 的查價功能，要做到什麼程度，才算能和 [FFXIV Market](https://github.com/beherw/FFXIV_Market) 媲美？

這裡的「媲美」不是指複製 UI，也不是照抄頁面結構，而是指以下四件事都要成立：

1. 使用者可以快速找到道具。
2. 找到道具後，可以立刻判斷值不值得買、賣、做。
3. 查價結果可以一路延伸到配方、取得方式、裝備挑選與歷史追蹤。
4. 這整套流程要能和 FF14 Helper 既有的製作頁、收藏頁工作流串起來。

FF14 Helper 目前的查價頁已經可用，但仍偏向「繁中服雙服工作表」。  
參考站則更接近「以單一道具為核心的市場資訊入口 + 一組衍生查價工具」。

本文件的目標，就是把這個落差寫清楚，並轉成可直接開發的規格。

## 3. 對齊定義

若要說 FF14 Helper 的查價功能已達到「可與參考站媲美」，至少要滿足以下條件：

### 3.1 搜尋入口不能只剩工作表

目前本站最強的是 OCR 匯入與雙服工作表。  
這對「我已經有材料清單」的玩家很好，但對「我只想先查一個道具」的玩家不夠直接。

對齊後必須同時具備：

- 單一道具搜尋
- OCR 截圖搜尋
- 批次清單匯入
- 歷史與推薦入口

### 3.2 單一道具查價必須是完整資訊面板

不能只有最低價與平均價。  
至少要讓使用者在同一個地方看到：

- 最低價
- 均價
- 最近購買價格
- 掛單清單
- 歷史交易
- 是否可交易
- 是否可製作
- 外部資料庫 / 取得方式入口
- 資料新鮮度

### 3.3 查價結果必須能延伸到實際決策

查價不只是看價格，而是支援決策：

- 要不要買
- 要不要自己做
- 要不要改去別服買
- 要不要用這個品項做主線裝備 / 製作裝備採購

### 3.4 必須保留 FF14 Helper 的站內整合優勢

對齊不是把參考站重做一份，而是把它的強項吸收到 FF14 Helper：

- 查價結果可送到試算與工作表
- 收藏頁任務可送到查價
- 製作頁配方材料可送到查價
- 整站維持前端優先與來源透明

## 4. 參考站架構摘要

FFXIV Market 的強，不在單一 API，而在它把多種市場工作流包成一個搜尋中心。

### 4.1 Search-first 架構

參考站以搜尋列為中心，再往外展開：

- 中文搜尋
- OCR 搜尋
- 搜尋歷史
- 物品歷史
- 相關關鍵字建議
- 相關物品延伸

這代表使用者第一步不是「建立工作表」，而是「先找到道具」。

### 4.2 單一道具市場面板

找到道具後，參考站會提供完整市場資訊：

- 資料中心或單服查詢切換
- 最低掛單
- 平均價格
- 最近購買
- 流通速度
- HQ / NQ 差異
- 在售清單
- 交易歷史
- 是否可交易

### 4.3 衍生查價工具

參考站不是只有一般查價，還有兩個很重要的衍生頁：

- 主線裝備查價器
- 製作職業裝備搜尋器

這代表它已經從「查一個道具」擴展到「解一個場景」。

### 4.4 資料層與效能策略

從參考站 repo 可以看出幾個關鍵做法：

- 本地 item / recipe / marketable 資料快取，不是每次都靠遠端搜尋。
- 批次查詢 Universalis，而不是逐筆串行請求。
- 先判斷 marketable，再決定是否進一步查市場資料。
- 對歷史、最近更新、裝備搜尋使用專門資料管線，而不是全塞在單一頁。
- 搜尋、伺服器切換、歷史載入都有 abort / retry / lazy load 處理。

## 5. FF14 Helper 目前現況

### 5.1 已有優勢

FF14 Helper 的查價頁並不弱，已有以下基礎：

- OCR 截圖匯入
- OCR 預覽校對
- 批次文字匯入
- 手動工作表輸入
- 陸行鳥 / 莫古力雙服比價
- 市場板利潤試算
- 批次道具查詢
- Garland Tools / 配方查詢入口
- 無後端、資料只留本機

### 5.2 核心落差

真正的差距主要有五類：

| 面向 | 參考站 | 本站目前狀態 | 結論 |
|------|--------|--------------|------|
| 入口 | 搜尋優先 + OCR + 歷史 + 最近更新 | 工作表優先，搜尋屬於附加 Tab | 需要重做資訊架構 |
| 單品深度 | 有完整市場面板 | 只有批次結果卡 | 需要單一道具頁 / 面板 |
| 市場情報 | 掛單、歷史、速度、tradeable、freshness | 只有最低價 / 均價的局部結果 | 需要補齊市場指標 |
| 衍生工具 | MSQ 裝備、製作裝備、歷史頁 | 尚未站內化 | 需要新增子頁或子模組 |
| 資料層 | 本地 item index + marketable 判斷 + cache | 多數查詢直接打 XIVAPI / Universalis | 需要建立本地索引與快取層 |

## 7. 目標產品定義

### 7.1 查價模組的新定位

FF14 Helper 的查價模組不應再只是「繁中服雙服工作表」，而應改成：

> 以單一道具搜尋為中心，兼容批次工作表、任務清單匯入、裝備採購與市場決策的站內查價中心。

### 7.2 建議資訊架構

建議保留目前 `/market` 主頁，但新增市場子頁：

- `/market`
  - 搜尋入口
  - OCR / 批次匯入
  - 工作表
  - 最近更新
- `/market/item/:itemId`
  - 單一道具市場資訊
- `/market/history`
  - 搜尋歷史 / 物品歷史
- `/market/msq`
  - 主線裝備查價器
- `/market/crafter-gear`
  - 製作職裝備查價器

如果短期不想擴張 route，也至少要在 `MarketPage` 內先做出：

- 主搜尋區
- item detail panel
- recent updates panel
- history panel

### 7.3 與現有 4 Tab 的關係

目前 4 Tab 不該直接丟掉，而是要重排優先序：

1. 搜尋 / OCR
2. 單一道具市場面板
3. 工作表
4. 試算
5. 最近更新 / 歷史
6. 模板 / 任務匯入

也就是說，工作表保留，但不再是唯一主流程。

## 8. 技術架構建議

### 8.1 資料層

建議新增 `src/market/` 模組層，避免所有邏輯繼續堆在 `MarketPage.tsx`：

```text
src/market/
  scopes.ts
  cache.ts
  searchIndex.ts
  history.ts
  templates.ts
  freshness.ts
  equipment.ts
  compare.ts
```

### 8.2 本地索引

要做到接近參考站的搜尋體驗，本站需要 build-time 本地索引，而不是每次用 XIVAPI 即時搜尋當主搜尋引擎。

建議新增：

- `public/market/items.tw.json` 或 msgpack
  - `itemId`
  - 繁中 / 簡中 / 英文名稱
  - icon id
  - patch / version
  - ilvl
  - 裝備部位
  - 製作職分類
  - marketable flag
- `public/market/marketable-items.json`
- `public/market/equipment-index.json`
- `public/market/recipe-index.json`

### 8.3 API 層擴充

`src/api/universalis.ts` 需要擴充，不只保留目前的單筆與批次查價：

- `fetchWorldMarket(itemId, world)`
- `fetchDcMarket(itemId, dc)`
- `fetchAggregatedMarkets(scope, itemIds)`
- `fetchRecentUpdates(scope, limit)`
- `fetchTaxRates(worldId or worldName)`
- `fetchScopeFreshness(scope)`

`src/api/xivapi.ts` 則應從「即時主搜尋」降級為：

- 補資料
- 配方搜尋
- 無法從本地索引命中時的 fallback

### 8.4 快取策略

至少需要兩層快取：

- 記憶體快取：同頁重查不重打 API
- `localStorage` TTL 快取：近期查過的 item detail、recent updates、history metadata

建議 TTL：

- item detail：60 秒
- recent updates：30 秒
- tax rates：5 分鐘
- equipment search results：10 分鐘

### 8.5 OCR 策略

本站目前已有 PaddleOCR + Tesseract fallback，這是優勢，不需要倒退。  
應繼續保留，但把 OCR 結果接到新的搜尋中心：

- 截圖中的第一個命中可直接開 item detail
- 多個命中可先進候選清單
- 候選清單可一鍵送入工作表

## 9. UI / UX 對齊規格

### 9.1 首屏必須同時支援兩種人

第一種使用者：

- 我只想查一個道具現在多少。

第二種使用者：

- 我手上有一批材料或一張市場板截圖，要快速整理。

因此首頁首屏至少要有：

- 主搜尋欄
- OCR / 貼上圖片入口
- scope selector
- 最近更新快捷
- 最近查過的道具

### 9.2 item detail 應包含的區塊

單一道具頁或詳情面板至少要有以下區塊：

1. 道具基本資訊
   - 名稱、icon、item id、可交易狀態、版本
2. 市場摘要卡
   - 最低價、均價、最近購買、流速、資料時間
3. 範圍切換
   - 資料中心 / 世界
4. 品質切換
   - 全部 / HQ / NQ
5. 在售清單
6. 交易歷史
7. 操作列
   - 加入工作表
   - 帶入試算
   - 查配方
   - 開 Garland / Universalis
8. 延伸資訊
   - 相關物品
   - 取得方式
   - 製作材料成本

### 9.3 recent updates 不應再使用「最近變更」概念混用

本站現在的「最近變更」只是在追蹤本機操作。  
這和參考站的「最近更新的物品」是兩件完全不同的事。

後續必須拆成：

- 本機操作紀錄
- 市場最近更新的物品

命名不能再混在一起。

### 9.4 scope selector 規格

FF14 Helper 不需要照抄參考站的所有伺服器策略，但至少要做到：

- 預設快速選項：`陸行鳥`、`莫古力`
- 進階模式：可切換對應資料中心或個別世界
- 顯示目前查詢粒度：世界或資料中心
- 切換 scope 時要清楚更新資料新鮮度與快取狀態

## 10. 與站內其他模組的整合

FF14 Helper 的優勢是多工具整合，這部分應刻意超過參考站。

### 10.1 與收藏頁整合

收藏頁可提供：

- 老主顧交件材料清單
- 友好部落任務相關材料清單

查價頁應支援：

- 一鍵從收藏頁帶入清單到工作表
- 依任務群組套用常用模板

### 10.2 與製作頁整合

製作頁可提供：

- 配方材料
- BOM 成本
- 製作職分類

查價頁應支援：

- 由配方一鍵帶入材料清單
- 對單一道具顯示「自己做 vs 直接買」的成本比較

### 10.3 與工具箱整合

工具箱中的 FFXIV Market 參考項目，之後應改成：

- 站內版查價：`partial` → `integrated`
- 外部參考連結仍保留，但說明本站已站內化哪些能力

## 11. 分階段執行計劃

### Phase 1：資料基礎與搜尋骨架

目標：把查價從即時 API 搜尋頁，提升成有本地索引的市場入口。

交付內容：

- 本地 item index builder
- marketable index
- scope registry
- 搜尋歷史 / 物品歷史儲存
- 主搜尋欄與 OCR 搜尋共用入口

完成標準：

- 中文搜尋不再完全依賴 XIVAPI 即時查詢
- 同一道具重查不會每次都重新打遠端 API

### Phase 2：單一道具市場詳情

目標：建立 item detail 核心體驗。

交付內容：

- `/market/item/:itemId` 或等價詳情面板
- 市場摘要卡
- 掛單清單
- 交易歷史
- scope 切換
- 資料新鮮度顯示

完成標準：

- 使用者查一個道具時，不需要回到工作表才能判斷行情

### Phase 3：市場情報補齊

目標：讓查價結果具備決策深度。

交付內容：

- HQ / NQ 分流
- 最近購買
- 流通速度
- 是否可交易
- 稅率 / 試算整合

完成標準：

- 能在單一道具頁完成「買 / 賣 / 做」初步判斷

### Phase 4：歷史與最近更新

目標：補齊搜尋中心的回訪能力。

交付內容：

- recent updates panel
- 搜尋歷史頁
- 物品歷史頁
- 常用快捷入口

完成標準：

- 使用者重回查價頁時，不必從空白狀態重新開始

### Phase 5：裝備查價工具

目標：做出與參考站同等級的場景型工具。

交付內容：

- 主線裝備查價器
- 製作職裝備查價器
- 裝備部位、ilvl、職業篩選

完成標準：

- 查價模組不再只服務材料，而能處理裝備採購場景

### Phase 6：站內整合超越點

目標：利用 FF14 Helper 的多頁優勢超過參考站。

交付內容：

- 收藏頁材料清單帶入
- 製作頁 BOM 帶入
- item detail 與 craft / collection 的互跳
- 常用模板

完成標準：

- 查價頁成為全站工作流中樞，而不是孤立工具頁

## 12. 可直接拆成 issue 的工作清單

### P0

## 14. 不做事項與邊界

為避免方向走偏，查價模組的對齊工作明確不包含：

- 複製參考站 UI、文案或視覺素材
- 依賴人工維護的私有後端
- 假裝存在官方繁中服即時市場 API
- 把查價頁做成全站唯一入口，反而破壞既有工具分工

## 15. 建議決策

### 15.1 不要再把查價頁只當成單頁工作表維護

如果繼續把所有功能塞進同一個 `MarketPage.tsx`，後續一定會失控。  
這次重構應該把市場資料層、搜尋層、歷史層、裝備層拆開。

### 15.2 優先建立本地索引，而不是先堆更多 UI

沒有本地 item index，就做不出接近參考站的搜尋手感。  
這是查價模組真正的 P0，不是裝飾。

### 15.3 超越參考站的路徑，不在 UI，而在整合

FF14 Helper 應該把優勢放在：

- 收藏任務 → 查價
- 配方材料 → 查價
- 單品詳情 → 試算 / 製作 / 外部資料庫

這才是站內整合專案的價值。
